# 质量保障体系

## Fake News 识别与处理

### 五层 Fake News 类型（必须删除）

**第一层：完全捏造**（最恶劣）
```
❌ "OpenClaw 在 2026 年 1 月从 Python 迁移到 Node.js"（实际从未发生）
❌ "OpenClaw 获得某知名投资机构投资"（纯属编造）
```

**第二层：张冠李戴**（混淆视听）
```
❌ "OpenClaw 采用了 XX 框架的架构"（实际毫无关系）
❌ "OpenClaw 是 XX 公司官方推出的工具"（实际是第三方项目）
```

**第三层：时间错乱**（误导认知）
```
❌ "昨天发布了新版本"（实际是一个月前）
❌ "即将在下个月支持 XX"（实际早已支持或不支持）
```

**第四层：数据造假**（量化欺骗）
```
❌ "拥有 100 万用户"（实际只有几万）
❌ "性能提升 10 倍"（无测试依据的夸大）
```

**第五层：断章取义**（选择性真实）
```
❌ "OpenClaw 不支持 Python"（实际是"不再推荐新用户使用"）
❌ "OpenClaw 完全免费"（实际是"开源免费，但 API 要钱"）
```

### 事实核查流程

1. 生成初稿后逐项核查
2. 重点关注：完全捏造的事件？张冠李戴的描述？编造的数据？
3. **处理原则**：
   - 互联网上有迹可循 → ✅ 保留
   - 无法证实但合理（经验估计、定性描述）→ ✅ 保留
   - 凭空捏造、无中生有 → ❌ 删除

## 发布前检查清单

### 真实性核查（必须通过）

- [ ] 事件真实性：所有提到的事件是否真实发生过？
- [ ] 归属正确性：是否将其他项目的特性错误归因？
- [ ] 时间准确性：所有时间描述是否与实际一致？
- [ ] 数据可靠性：所有具体数字是否有测试依据？
- [ ] 表述完整性：是否有断章取义导致误导？

### 内容完整性（必须满足）

- [ ] 有明确的实操步骤（1-2-3 列表）
- [ ] 有可运行的代码片段
- [ ] 有预期结果说明
- [ ] 有常见问题及解决方案
- [ ] 有配套插图/示意图

### 平台适配性（公众号专享）

- [ ] 标题 20-26 字（最优长度）
- [ ] 摘要前 50 字吸引人（决定打开率）
- [ ] 段落不超过 5 行（阅读友好）
- [ ] 每 500-800 字有小标题
- [ ] 重点句子加粗突出
- [ ] 有完整的行动号召
- [ ] 封面图已生成且与主题相关
- [ ] 配图清晰且相关

### 禁止出现的内容

- 纯理论阐述 without 代码
- 空泛的"应该怎么做"
- 没有验证路径的结论
- 未经证实的技术参数
- 来源不明的"据说"、"网传"
- 低质量或与主题无关的配图
